# Parity Validation

Parity tests compare Ocelotl output with a trusted reference or with another
Ocelotl backend. They are the main guardrail for CPU, GPU, cache, and
quantization work.

## Parity Types

- Loader parity: parsed metadata matches expected fixture metadata.
- Tokenizer parity: token IDs match a known tokenizer implementation.
- CPU reference parity: CPU output matches committed fixtures or a trusted
  external reference.
- GPU parity: GPU output matches CPU within tolerance.
- Cache parity: contiguous and paged KV produce equivalent logits or tokens.
- Quantization parity: quantized output stays within documented limits.

## Reference Selection

A reference must be named explicitly. Examples include:

- A committed fixture generated by a known tool and model revision.
- A small hand-constructed tensor case with exact expected output.
- A trusted external runtime used only to generate fixture data.

Do not compare against a moving online model endpoint.

## Tolerance Policy

Each parity test must document its tolerance. Exact token equality is preferred
for deterministic greedy tests. Floating-point tolerances should be as tight as
practical and justified when widened.

## Required Before Defaulting

A new backend, cache layout, quantization format, or model family should not
become default until it has parity coverage for at least:

- A short prompt.
- A prompt near a cache boundary when cache behavior is involved.
- An unsupported-config failure test.

## M3 — Qwen2.5 Forward Path

M3 parity is split into two default-on checks:

- **Prefill logits parity** compares final-position logits against the
  committed tiny-synthetic fixture within a documented floating-point
  tolerance.
- **One-token decode parity** derives the selected token from the prefill
  fixture and asserts exact token equality through the public runtime decode
  path.

The comparison sources and tolerances below are part of the M3 contract. A
change to the fixture tolerance, decode token, or comparison source must update
this document and the corresponding tests in the same patch.

### Reference

- **Tiny synthetic prefill, default-on**:
  `fixtures/logits/qwen2_5_tiny_synthetic_prefill.json` is the offline
  parity fixture for M3.7. Generated by running
  `Qwen2_5Model::prefill` against the tiny Qwen2.5-shaped model
  (vocab=32, hidden=16, intermediate=32, num_hidden_layers=2,
  num_attention_heads=4, num_key_value_heads=2, head_dim=4) with
  deterministic synthetic weights. Source: this repository, this commit.
  The fixture pins the final-position logits over the vocabulary;
  reproducibility is the contract. The default-on test is
  `crates/models/tests/qwen2_5_tiny_synthetic_prefill.rs`.
- **Tiny synthetic one-token decode, default-on**:
  `crates/runtime/tests/qwen2_5_tiny_synthetic_decode.rs` runs
  `ocelotl_runtime::decode_one_token` against the same model shape, prompt
  `[3, 7, 11]`, and deterministic synthetic weights as the M3.7 prefill
  fixture. The expected token is `TokenId(16)`, derived from the strict argmax
  of the M3.7 fixture logits plus the M1.8 greedy-sampling contract. Decode
  parity is exact token equality; it has no numeric tolerance.
- **Real Qwen2.5-0.5B-Instruct, opt-in (deferred to a follow-up)**:
  HuggingFace `Qwen/Qwen2.5-0.5B-Instruct` at SHA
  `7ae557604adf67be50417f59c2c2f167def9a775`. Captured logits should be
  produced by HuggingFace `transformers` running in `f32` (not the
  ship-default `bf16`) so the comparison surface matches Ocelotl's CPU
  reference path. The opt-in test should follow the M2.3
  default-plus-ignored semantic pattern: default-on fixture-shape validation,
  `#[ignore]` for the local-artifact semantic check, and a clear skip message
  naming `docs/artifact-preparation.md`. The capture script must pin the
  Python package versions, call `model.float()` explicitly, run with
  `torch.no_grad()`, and record the exact prompt, tokenizer/config revision,
  dtype, and generation command in the fixture. Shipping this depends on a
  `Qwen2_5Weights::from_safetensors` weight loader, so it is deferred from M3.

### Tolerance

| Fixture | Tolerance | Justification |
| --- | --- | --- |
| `qwen2_5_tiny_synthetic_prefill` (M3.7) | `1e-4` | Two-layer prefill chain, k=16 matmuls, attention reduction ~32 elements, MLP intermediate=32. Empirical worst-case observed in M3.5 attention is ~2e-6; depth multiplier + cross-platform `f32::sin/cos/exp` drift gives an estimated ~5e-5 worst case. 1e-4 leaves an order of magnitude headroom and still catches a logic bug (typically >>1e-3 movement). See test module docs at `crates/models/tests/qwen2_5_tiny_synthetic_prefill.rs` for the full derivation. |
| `qwen2_5_tiny_synthetic_decode` (M3.8) | exact token equality (`TokenId(16)`) | Decode is `runtime::prefill` plus greedy argmax. The prefill numeric tolerance is already enforced by the M3.7 fixture test; decode parity only needs to prove the public runtime path selects the deterministic argmax token. |
| Real Qwen2.5-0.5B-Instruct (deferred) | `1e-3` (target) | 24-layer chain at hidden=896 / intermediate=4864. The depth multiplier on M3.5's empirical 2e-6 base puts the worst-case at ~1e-4; widening to 1e-3 absorbs HF transformers' own `f32` rounding differences (HF defaults to bf16; an explicit `model.float()` is needed and itself drifts from training-time bf16 values). Ratify or tighten after the first captured real-model fixture reports observed max absolute error. |

### Determinism

Greedy sampling is exact-token equality. Logits parity is the f32
floating-point comparison documented above. Both must hold.

### Policy Tripwires

- `crates/models/tests/qwen2_5_tiny_synthetic_prefill.rs` asserts that the
  JSON fixture tolerance equals the test's `TOLERANCE` constant before comparing
  per-element logits.
- `crates/runtime/tests/qwen2_5_tiny_synthetic_decode.rs` asserts the selected
  token is exactly `TokenId(16)` through `ocelotl_runtime::decode_one_token`.
- `crates/models/tests/qwen2_5_m3_parity_policy.rs` asserts this document names
  the M3 fixture, the prefill tolerance, the decode test, the exact decode
  token, and the deferred real-Qwen target tolerance. This is the M3.9
  tolerance/doc update requirement.

### Deferred Follow-ups

- Add a hash-of-logits sanity field only if fixture review becomes noisy. The
  current default-on fixture has 32 logits, so the per-element vector remains
  the authoritative comparison source for M3.
- Add the real-Qwen opt-in parity fixture after `Qwen2_5Weights::from_safetensors`
  exists. The first implementation should commit the capture script, captured
  metadata, observed max absolute error, and any adjustment to the target
  `1e-3` tolerance in one patch.

## M4 — CubeCL WGPU Kernel Path

M4 GPU parity is CPU-referenced at every level. The CubeCL path must not compare
against previously captured GPU output.

- **RoPE kernel parity, opt-in local execution**:
  `ocelotl_kernels::cubecl_backend::tests::wgpu_rope_matches_cpu_reference_for_position_one`
  runs the CubeCL WGPU RoPE kernel and compares against
  `ocelotl_kernels::rope_apply_inplace` at `1e-5`.
- **Qwen prefill parity, opt-in local execution**:
  `crates/models/tests/qwen2_5_tiny_synthetic_prefill.rs::cubecl_wgpu_prefill_matches_cpu_reference_within_tolerance`
  constructs a Qwen model with `Qwen2_5KernelBackend::CubeClWgpu`, runs the M3
  tiny prompt, and compares logits against the CPU model at the existing M3
  `1e-4` tolerance.
- **Qwen decode parity, opt-in local execution**:
  `crates/runtime/tests/qwen2_5_tiny_synthetic_decode.rs::cubecl_wgpu_decode_one_token_matches_cpu_reference`
  constructs the CubeCL-backed model through `Runtime::cubecl_wgpu(0)` and
  asserts exact token equality against CPU decode.

The M4 model path is intentionally partial: RoPE uses CubeCL WGPU, and matmul,
attention, RMSNorm, MLP, residual adds, and logits projection use CPU fallback.
Full-model GPU execution remains deferred.

## Post-M3 Whisper ASR

Whisper ASR parity is split into default-on synthetic coverage and an opt-in
local-artifact harness:

- **Tiny synthetic Whisper path, default-on**:
  `crates/models/tests/whisper_tiny_synthetic.rs` proves the W-ASR.4
  encoder/decoder wiring shape without network or local artifacts.
- **Whisper tiny.en local bundle, opt-in**:
  `crates/models/tests/whisper_local_artifact_parity.rs` documents and checks
  `local-artifacts/whisper_tiny_en/{config.json,tokenizer.json,model.safetensors,reference/sample_16khz_mono.wav,reference/expected_tokens.json}`.
  W-ASR.15 adds optional timestamped parity through
  `reference/expected_tokens_timestamped.json`.
  The default-on tests in that file validate the expected token-reference
  schema, including supported no-timestamps and timestamped startup variants.
  The ignored no-timestamps test validates the local bundle exists, parses
  `config.json`, `tokenizer.json`, and `expected_tokens.json`, derives the
  tokenizer/model family from `vocab_size`, validates the safetensors manifest,
  loads required tensor values, decodes PCM16 or IEEE float32 WAV sample data,
  computes log-mel features, and compares exact autoregressively generated
  token IDs against the reference sequence. The ignored timestamped test skips
  clearly when `expected_tokens_timestamped.json` is absent; when present, it
  runs the same local bundle with timestamp tokens enabled and compares exact
  token IDs plus parsed segment boundaries.

### Reference

The opt-in Whisper reference is `reference/expected_tokens.json` inside the
local bundle. It must name the sample audio path, use task `transcribe`,
language `en`, `timestamps = false`, and carry an `expected_token_ids` sequence
that starts with the no-timestamps startup prompt appropriate to the artifact's
tokenizer/model family and includes at least one generated token after the
startup prompt. W-ASR.15 also recognizes an optional
`reference/expected_tokens_timestamped.json` file with `timestamps = true`,
timestamp boundary token IDs in `expected_token_ids`, and `expected_segments`
that must exactly match the public tokenizer timestamp-segment parser. Current
supported variants are:

| Artifact family | Condition | Startup prompt | EOT | First timestamp |
| --- | --- | --- | --- | --- |
| OpenAI English-only Whisper | `vocab_size < 51865` | `[50257, 50362]` | `50256` | `50363` |
| OpenAI multilingual Whisper, English transcribe | `vocab_size >= 51865` | `[50258, 50259, 50359, 50363]` | `50257` | `50364` |

Timestamped references omit `<|notimestamps|>` from the startup prompt:

| Artifact family | Timestamped startup prompt |
| --- | --- |
| OpenAI English-only Whisper | `[50257]` |
| OpenAI multilingual Whisper, English transcribe | `[50258, 50259, 50359]` |

Exact token equality is the parity contract for the opt-in local bundle.
Timestamped parity additionally requires exact segment-boundary equality after
parsing the generated suffix.

### Current Limit

W-ASR.5 proves the local-artifact contract and reference shape. W-ASR.7 adds
generic safetensors tensor-value loading, W-ASR.8 adds a real Whisper
config/tensor-manifest contract, W-ASR.9 adds the real CPU adapter, and W-ASR.10
extends the ignored harness from "bundle is well-formed" to "output tokens equal
expected tokens". W-ASR.15 keeps that proof intact and layers timestamped
schema/default tests plus an optional timestamped ignored proof on top.

The current branch does not include `local-artifacts/whisper_tiny_en`, so the
exact local parity proof is still artifact-blocked until a contributor provides
that bundle and runs the ignored test. The tensor contract uses OpenAI Whisper
state-dict names as the canonical Ocelotl contract; aliases for HF/Burn-converted
safetensors names should be added only after a local `model.safetensors`
manifest proves the needed alternate names.
