# Parity Validation

Parity tests compare Ocelotl output with a trusted reference or with another
Ocelotl backend. They are the main guardrail for CPU, GPU, cache, and
quantization work.

## Parity Types

- Loader parity: parsed metadata matches expected fixture metadata.
- Tokenizer parity: token IDs match a known tokenizer implementation.
- CPU reference parity: CPU output matches committed fixtures or a trusted
  external reference.
- GPU parity: GPU output matches CPU within tolerance.
- Cache parity: contiguous and paged KV produce equivalent logits or tokens.
- Quantization parity: quantized output stays within documented limits.

## Reference Selection

A reference must be named explicitly. Examples include:

- A committed fixture generated by a known tool and model revision.
- A small hand-constructed tensor case with exact expected output.
- A trusted external runtime used only to generate fixture data.

Do not compare against a moving online model endpoint.

## Tolerance Policy

Each parity test must document its tolerance. Exact token equality is preferred
for deterministic greedy tests. Floating-point tolerances should be as tight as
practical and justified when widened.

## Required Before Defaulting

A new backend, cache layout, quantization format, or model family should not
become default until it has parity coverage for at least:

- A short prompt.
- A prompt near a cache boundary when cache behavior is involved.
- An unsupported-config failure test.

## M3 — Qwen2.5 Forward Path

This section seeds the M3.9 formalization. M3.7 (prefill) and M3.8 (decode)
are the producers; M3.9 will tighten and freeze the rules.

### Reference

- **Tiny synthetic, default-on**:
  `fixtures/logits/qwen2_5_tiny_synthetic_prefill.json` is the offline
  parity fixture for M3.7. Generated by running
  `Qwen2_5Model::prefill` against the tiny Qwen2.5-shaped model
  (vocab=32, hidden=16, intermediate=32, num_hidden_layers=2,
  num_attention_heads=4, num_key_value_heads=2, head_dim=4) with
  deterministic synthetic weights. Source: this repository, this commit.
  The fixture pins the final-position logits over the vocabulary;
  reproducibility is the contract.
- **Real Qwen2.5-0.5B-Instruct, opt-in (deferred to a follow-up)**:
  HuggingFace `Qwen/Qwen2.5-0.5B-Instruct` at SHA
  `7ae557604adf67be50417f59c2c2f167def9a775`. Captured logits should be
  produced by HuggingFace `transformers` running in `f32` (not the
  ship-default `bf16`) so the comparison surface matches Ocelotl's
  reference path. The capture script and the gating mechanism (the
  `#[ignore]` default-plus-ignored pattern from M2.3) are not yet
  shipped; introducing them is a small follow-up that depends on a
  `Qwen2_5Weights::from_safetensors` weight loader. The M3.7 brief
  recommended both default-on and opt-in; only default-on shipped in
  M3.7. M3.9 should formalize the opt-in flow.

### Tolerance

| Fixture | Tolerance | Justification |
| --- | --- | --- |
| `qwen2_5_tiny_synthetic_prefill` (M3.7) | `1e-4` | Two-layer prefill chain, k=16 matmuls, attention reduction ~32 elements, MLP intermediate=32. Empirical worst-case observed in M3.5 attention is ~2e-6; depth multiplier + cross-platform `f32::sin/cos/exp` drift gives an estimated ~5e-5 worst case. 1e-4 leaves an order of magnitude headroom and still catches a logic bug (typically >>1e-3 movement). See test module docs at `crates/models/tests/qwen2_5_tiny_synthetic_prefill.rs` for the full derivation. |
| Real Qwen2.5-0.5B-Instruct (M3.7, deferred) | `1e-3` (target) | 24-layer chain at hidden=896 / intermediate=4864. The depth multiplier on M3.5's empirical 2e-6 base puts the worst-case at ~1e-4; widening to 1e-3 absorbs HF transformers' own `f32` rounding differences (HF defaults to bf16; an explicit `model.float()` is needed and itself drifts from training-time bf16 values). |

### Determinism

Greedy sampling is exact-token equality. Logits parity is the f32
floating-point comparison documented above. Both must hold.

### Open formalization items for M3.9

- Whether to add a hash-of-logits sanity field alongside the per-element
  vector (cheaper to diff in PR review than 32+ floats).
- Whether the tolerance lives in the fixture (current M3.7 choice) or
  in code (current M1.7+ kernel-test choice). The fixture-side approach
  was chosen for M3.7 because it makes a tolerance loosening visible on
  review; the kernel approach was chosen earlier because the kernel
  contract is more stable than the fixture contents. Both are
  defensible; pick one explicitly in M3.9.
- The exact `transformers` capture script (Python + version + dtype)
  for the real-Qwen parity test, when it lands.
